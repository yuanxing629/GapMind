# GapMind PRD：RAG、知识图谱、微调的作用和 Phase 3 任务依赖

> 这份文档讲三件事：RAG、知识图谱、微调在我们这个项目里是干嘛的，以及 Phase 3 阶段三个人的任务怎么排。
> 写作时间：2026 年 7 月。读者：团队全员。状态：PRD 阶段（讲清楚要做什么、为什么、给谁，不是设计到表和字段）。

---

## 一句话回答：这三个东西在我们项目里各自干嘛

RAG 是"找"的。当系统想从几十几百篇论文里找出跟某个问题相关的段落时，它负责搜索。它把要查的内容转成向量，去向量数据库里找最像的段落。

知识图谱是"存"的。RAG 找回来的段落是散落的文本片段，知识图谱把这些片段里的实体和实体之间的关系（比如某篇论文提出了什么方法、谁反驳了谁）系统地存起来，让系统知道这些段落"说的是什么"。

微调是"换"的。我们直接用 Deepseek 这种通用大模型来做知识抽取，每次都要花钱而且结果不一定稳定。微调就是用我们自己领域的论文数据，训练一个更小但更专业的模型，让它专门做这件事，把 Deepseek 替换掉。

---

## 详细讲一下

### RAG（检索增强生成）

RAG 的本质是"先搜索，再让大模型综合答案"。具体流程是：用户问一个问题，系统把问题转成向量，去向量数据库里找出最相似的几段文本，然后把这些文本连同问题一起喂给大模型，让大模型基于这些文本综合出答案。

我们项目里 RAG 用来做三件事。

第一件事叫"similar work 检索"，给定一篇论文的全文，搜出跟它相似工作最相关的其他论文片段。这对 Discover Agent 找类似研究机会特别重要。

第二件事叫"counter-evidence 检索"，给定一条 claim 主张，搜出反驳或者限定这条 claim 的证据。这对 Discover Agent 判断一个研究机会是否真有价值特别重要。

第三件事是一般的语义搜索，用户在前端输入自然语言查询，系统从已上传的所有论文里找出相关片段。

我们项目里 RAG 的特别之处是它会结合知识图谱做"GraphRAG"增强。普通的 RAG 只是找相似文本片段，GraphRAG 会先在知识图谱里找到相关的实体节点，然后顺着这些节点的关系去找更多相关的证据。这样的好处是检索的精确度更高，相关的证据更全。

RAG 的工作由 zwx 负责，他要搭 Milvus 向量数据库，实现这三个检索函数。Phase 3 阶段他就要开始做。

### 知识图谱

知识图谱的本质是把"实体"和"实体之间的关系"存成图。实体可以是一篇论文、一种方法、一个任务、一个数据集、一条 claim 主张、一个 limitation 局限，关系可以是"论文 A 提出了方法 B""论文 A 解决了任务 C""论文 B 反驳了论文 A 的某条 claim"。

我们项目里知识图谱用来做四件事。

第一件事是结构化展示。前端用图谱可视化展示论文、方法和它们之间的关联关系，让用户能直观看到研究领域的全貌。

第二件事是支撑 RAG。GraphRAG 在向量检索时利用知识图谱的关系，让检索更精准。

第三件事是支撑 Discover Agent。系统化地分析哪些方法存在、哪些 claim 有反证、哪些研究方向还没被覆盖。

第四件事是支撑实验分析。找相关实验时基于知识图谱中的关联，比纯文本检索更准确。

我们项目里知识图谱的特殊之处是用三个表（knowledge_items、knowledge_relations、evidence_spans）来存，不用专门的图数据库（像 Neo4j 那种）。因为 MVP 阶段我们只需要查邻居节点（"找某个方法的所有相关工作"），不用做复杂的图遍历，关系表加索引就够了。

知识图谱的内容不是手工构建的，是用大模型从论文里抽取出来的。Phase 3 阶段，yx 用 Deepseek 从每篇论文的全文里抽取实体和关系，存进知识图谱。

### 微调

微调的本质是在预训练大模型的基础上，用特定领域的数据继续训练，得到一个领域专用的小模型。

我们项目里微调用来做两件事。

第一件事是"抽取微调"。训练样本是"论文全文 + 标注好的知识条目"对，训练后的小模型输入论文全文，输出知识条目的 JSON 格式。这样每次抽取都调用自己的小模型，不用花 Deepseek 的钱。

第二件事是"Opportunity 生成微调"（Phase 5 之后）。训练样本是"证据 + 类似工作 + 反证 + 高质量研究机会"对，训练后的小模型输入证据，输出研究机会的描述。

我们项目里用微调的原因有四个。第一是成本——每次抽取用 Deepseek 大概要花一两毛钱，30 到 40 篇论文 1-2 千次调用就是几百块。微调后用自家的小模型推理成本几乎为零。第二是速度——Deepseek 每次一两秒，自家小模型零几秒。第三是稳定性——通用模型输出 JSON 有时不合规，训练时强 schema 约束后输出很稳定。第四是领域准确性——专门在 GNN 领域微调的模型对学术细节的把握比通用模型好。

我们项目里微调不是必需的。Phase 3 初期用 Deepseek 就能跑通，微调是"优化"——质量更高、成本更低、速度更快。zf 负责微调工作，用 Qwen3:8B 做基座，QLoRA 方法训练，单张 3090 显卡就够。

## 三者怎么配合

简单说，三者不是替代关系，是互补关系。

RAG 负责"找"。它从大量文本里找到相关的段落。
知识图谱负责"存"。它把 RAG 找回来的段落里的实体和关系系统化存起来。
微调负责"换"。它把抽取这个环节用的大模型替换成自家的小模型。

一个研究者的实际工作流是这样的。

第一步是找研究机会。用户输入研究主题和关键词，系统用 RAG 从已上传的论文里找相关内容，同时用知识图谱帮他看哪里有空白。这个过程输出若干研究机会的候选。

第二步是验证机会是否新颖。系统用 RAG 找类似工作，找反证，看是不是真的没人做过、是不是真的值得做。

第三步是转成可执行的研究计划。系统用知识图谱里已有的方法、任务、数据集关系，结合 LLM（微调后的小模型）生成具体的研究问题、假设、验证方案。

第四步是用户跑实验。手动做。

第五步是分析结果。系统用 RAG 找类似实验结果，对比分析。

可以看到，RAG 贯穿全流程，第一步、第二步、第五步都要"找证据"。知识图谱是底层结构，所有步骤都有知识图谱支撑。微调只在一个地方用，就是在抽取实体和关系时替代 Deepseek。

具体到技术实现上，PDF 解析之后会产出两个产物：parsed_text（论文全文）和 chunks（分块后的检索单位）。知识图谱的抽取用 parsed_text 作为输入，RAG 检索用 chunks 作为输入。两者完全解耦，parsed_text 不会进向量数据库，chunks 不会进大模型。

RAG 借助知识图谱来增强检索（GraphRAG），知识图谱的内容来自大模型抽取（未来可以来自微调后的小模型）。这就是 RAG 和知识图谱的关系。

## Phase 3 任务依赖

Phase 3 阶段我们要做三件事。一是让 yx 用 Deepseek 把知识抽取跑通（这是基线，所有后续工作的基础）。二是让 zwx 把 RAG 检索系统搭起来（Milvus 索引 + 三个检索函数）。三是让 zf 准备微调训练数据并开始训练。

这三件事不是完全独立的，有先后顺序。让我用一段话讲清楚依赖关系。

**最先做的是 yx 的知识抽取基线**。原因很简单——所有其他工作都依赖知识图谱里有内容。zwx 训练 GraphRAG 增强需要知识图谱有 entity 和 relation，zf 训练模型需要 yx 跑出来的 KnowledgeItem 作为 ground truth 标注。所以 yx 必须先把抽取 pipeline 跑通，至少能跑出 30 到 40 篇论文的初始 KnowledgeItem。这是 Phase 3 第一周的目标。

**第二件做的是 zf 的训练数据准备**。一旦 yx 有 KnowledgeItem 产出，zf 就可以开始了。zf 的工作分两半：一半是构造训练样本（用 yx 抽出来的 KnowledgeItem 作为 ground truth 标注，再由 zf 和 yx 一起人工修正为 gold 标准），另一半是用 Qwen3:8B 做微调训练。训练本身不依赖 yx 的其他工作（Deepseek 抽取已经能跑），但训练数据依赖 yx 的 KnowledgeItem 产出。这是 Phase 3 第二到第三周的工作。

**第三件做的是 zwx 的 RAG 检索系统**。zwx 工作的输入是 chunks（PDF 解析的产物，这个已经在 Phase 2 完成），不依赖知识图谱。理论上 zwx 可以和 yx 的抽取工作并行。但 zwx 真正能用上"知识图谱增强检索"功能，需要 yx 已经有 KnowledgeItem 产出。所以 zwx 的工作是"先搭基础（Milvus 索引 + 基本检索）"，再后续"加 GraphRAG 增强"。这是 Phase 3 第二到第四周的工作。

**有一个隐含的第四项是 yx 做的 Discover Agent**。但这是 Phase 5 的内容，不在 Phase 3 范围。

下面用文字描述 Phase 3 周计划。

第一周，yx 搭抽取 pipeline（用 Deepseek），跑 Smoke Set（8 到 12 篇论文）的初始抽取，把 5 种类型的 KnowledgeItem 都跑出来。zwx 同时开始搭 Milvus 基础（创建 collection、定义 schema），不依赖 yx。

第二周，yx 继续跑更多论文（向 30 到 40 篇目标推进）。zwx 完成 Milvus 索引脚本，开始实现三个检索函数（semantic_search、find_similar_work、find_counter_evidence）。zf 开始看 yx 抽出来的 KnowledgeItem，了解领域知识，着手准备训练数据格式。

第三周，yx 跑完 30 到 40 篇论文的抽取。zwx 三个检索函数都能跑通了，开始构造评估数据集（20-30 个 query + ground truth）。zf 开始构造 silver 训练样本（用 yx 抽出来的 KnowledgeItem 当 ground truth）。

第四周，zwx 跑评估基线指标（Recall、MRR 等）。zf 和 yx 一起做 gold 标注（从 silver 抽 300-500 个修正为 gold），开始微调训练。

第五到六周，zf 跑 LoRA 训练，zwx 优化检索（可能加 GraphRAG 增强）。

第六到七周，zf 部署微调模型（vLLM 暴露 OpenAI 兼容 API），yx 把微调模型接入 Model Gateway。zwx 完成 GraphRAG 增强检索。

整体来说，yx 的知识抽取是 Phase 3 的关键路径，所有人都依赖它。zwx 的 RAG 系统可以晚一些（第三周后才开始有完整可用的产出）。zf 的微调是优化项，不影响主线（用 Deepseek 也能跑通 Phase 3 全部功能）。

## 同步点

每周末一次同步会，30 到 60 分钟，过一下这周产出和下周计划。

第一周末，yx 给 zwx 展示抽取结果，让 zwx 知道 KnowledgeItem 长什么样、relation 怎么定义的。

第二周末，yx 跑出 20+ 篇论文的抽取。zwx 第一个检索函数跑通。zf 看了 yx 抽取结果，定了训练数据 schema。

第三周末，yx 完成 30+ 篇论文抽取。zwx 三个检索函数都跑通。zf silver 训练样本构造好。

第四周末，zwx 跑出 RAG 评估指标。zf gold 标注完成（300+ 条），开始微调训练。

第五到六周，zf 跑训练、调参。zwx 优化检索。

第七周，zf 部署微调模型，端到端测试（用微调模型抽一篇新论文，存 KnowledgeItem，建 Milvus 索引，用 RAG 检索）。

## 一些边界说明

**RAG 不进 LLM 做抽取**。RAG 的输出是 chunks 片段，给 LLM 综合答案用，不是给 LLM 抽取实体用。这两个用法是分开的。

**抽取的输入是论文全文不是 chunks**。论文里大部分 chunk 段没有有效的结构化知识，chunk 级抽取会浪费 token 而且切断上下文。LLM 需要看整篇论文才能正确判断实体之间的关系。

**微调不是必需**。Phase 3 初期用 Deepseek 就能跑通所有功能，微调是优化项（成本低、速度快、稳定性高）。如果没有 GPU 资源或时间不够，Phase 3 完全可以不微调直接交付。

**知识图谱是逻辑结构不是物理图数据库**。我们用三个关系表来存图结构，不用 Neo4j 这种专门图数据库。

---

## 总结

RAG 在我们项目里是"找"——找证据、找类似工作、找反证。

知识图谱是"存"——存实体、存关系、让系统知道这些证据在说什么。

微调是"换"——把 Deepseek 这种通用大模型换成自家在领域论文上微调过的小模型。

Phase 3 阶段，yx 的知识抽取是关键路径（所有人依赖）。zwx 的 RAG 系统可以晚一两周启动。zf 的微调是优化项不阻塞主线。

三个技术合起来，就是 GapMind 旗舰能力"基于证据的研究机会发现"的完整技术栈。
